## Visual iteration protocol (mandatory for every plan in this suite)

All three plans change what the user SEES. Tests prove the contract; they say
nothing about whether a crown reads as cozy or a marker reads as intentional.
Every visual-affecting step therefore runs this loop — budget for it; one-shot
visual changes in this codebase have consistently needed 2–4 rounds (the
2026-07-09 palm remodel took four).

**The loop**: capture baseline → make the change → capture again → critique
against the step's named criteria (each plan lists them per step) → tweak →
re-capture. Exit only when every criterion is met in ALL required views. Never
judge from the code or from a single angle.

**Tooling** — the `agent-browser` CLI (assume installed; `agent-browser open`
fails otherwise → install with `npm i -g agent-browser && agent-browser install`).
Proven command sequences against this app (from the palm-remodel session):

```bash
# Serve (port 5180; vite bumps to 5181 if occupied — read the dev output!)
pnpm dev:editor

# Model gallery (all kinds × 3 seeds, theme switcher):
agent-browser open "http://localhost:5180/?gallery" && sleep 3

# Switch texture theme (refs from snapshot: Classic/Pastel/Storybook/Off):
agent-browser snapshot -i          # lists @eN refs for the theme buttons
agent-browser click @e3            # e.g. Pastel

# Zoom in (OrbitControls wheel; the gallery starts far out):
agent-browser eval "(()=>{const cv=document.querySelector('canvas'); const r=cv.getBoundingClientRect(); for(let i=0;i<12;i++){cv.dispatchEvent(new WheelEvent('wheel',{deltaY:-120,clientX:r.width/2,clientY:r.height/2,bubbles:true,cancelable:true}))} return 'zoomed'})()"

# Orbit to top-down (drag down = camera up) / back (drag up):
agent-browser eval "(()=>{const cv=document.querySelector('canvas'); const r=cv.getBoundingClientRect(); const x0=r.width/2,y0=r.height/2; cv.dispatchEvent(new PointerEvent('pointerdown',{clientX:x0,clientY:y0,button:0,buttons:1,pointerId:5,bubbles:true})); for(let i=1;i<=8;i++){cv.dispatchEvent(new PointerEvent('pointermove',{clientX:x0,clientY:y0+i*20,button:0,buttons:1,pointerId:5,bubbles:true}))} cv.dispatchEvent(new PointerEvent('pointerup',{clientX:x0,clientY:y0+160,button:0,pointerId:5,bubbles:true})); return 'top'})()"

agent-browser screenshot /tmp/<plan>-<step>-<view>.png
```

Gotchas learned the hard way:
- The gallery ships live WIND — crowns lean differently between captures.
  Don't chase per-frame differences; for wind-motion checks, capture 2–3
  frames ~2 s apart and diff them (difference proves wind works; identical
  frames at amp > 0 mean it doesn't).
- When simulating cmd-gestures in the main editor, set `metaKey: true` on the
  POINTER events, not just a Meta keydown — three.js OrbitControls reads the
  pointer event's modifier flags (a keydown-only simulation false-passed a
  broken orbit feature once already).
- Standard views per capture round: **side (default), top-down, eye-level** —
  the game camera is high, so top-down is the view users actually see most.
- Judge at GAME zoom, not just gallery close-up: also load the main editor
  (`http://localhost:5180/`), place the object on an island, and screenshot.

**Subjective criteria tie-breaker**: some per-step criteria are aesthetic by
nature ("reads as grass", "layered not lumpy"). When a round leaves you
uncertain whether a criterion is met, do not self-certify — send the capture
set to the operator and ask; their call is final.

**Reporting**: keep every round's screenshots (`/tmp/<plan>-round<N>-*.png`)
and reference the final set in the commit/PR body. A plan's status row may
only move to DONE with the final capture set taken AFTER the last code change.

## Findings considered and rejected

- **GLSL/instanced wind for grass now** — rejected for 001: per-placement
  `CanopySpring` already scales to current object counts and keeps wind
  single-source with zero new code; instancing is the recorded follow-up if a
  scatter brush ships hundreds of tufts.
- **Base UI dependency for the contextual dropdown** — rejected: the
  island-editor workspace is deliberately dependency-light and hand-rolls its
  panels; one menu doesn't justify a new dependency (the product-app Base UI
  rule applies to `src/`, not this isolated workspace).
- **Removing the classic hotbar in 003** — rejected for this iteration:
  water/path/erase have no contextual gesture yet; the classic mode is the
  fallback and the regression baseline.
